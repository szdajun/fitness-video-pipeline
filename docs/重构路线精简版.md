# 健身视频优化流水线重构路线精简版

## 文档目的

这份文档用于沉淀当前阶段最值得投入的重构方向。目标不是做一轮“全面框架化重构”，而是基于项目已经跑通、效果优先、迭代速度重要的现实，筛出少数高收益改造项。

当前结论已经比较明确：

- 立即推荐：`Manifest 增量恢复`
- 立即推荐：`H2V 策略拆分`
- 顺手补做：`轻量 Metrics`
- 同步整理：`更新过时文档`、`整理根目录与 .gitignore`
- 暂缓：`配置大一统`、`Stage 契约体系`、`Context 分层`

---

## 当前最实际的三个问题

### 1. 缓存恢复机制脆弱

当前 engine 主要通过扫描输出目录和猜测文件名来恢复中间产物，这在以下场景容易出问题：

- 中间文件命名变化
- 出现 `*_new.mp4` 这类 fallback 文件
- 配置改了但旧缓存被误复用
- 输出目录混入历史残留文件

这类问题会直接影响：

- 增量处理稳定性
- 排障效率
- 重复运行耗时

### 2. `03_h2v_convert.py` 过重

当前 `03_h2v_convert.py` 同时承担了：

1. 输入读取
2. 人物 track 构建
3. 领操人识别
4. 场景判定
5. 分段合并
6. FFmpeg 裁切
7. cropped keypoints 回写

这不是“写错了”，而是职责太多。结果就是：

- 想改领操人识别，要碰大文件
- 想调全景判定，很难快速定位
- 想做针对性测试，不够方便

### 3. 缺少结构化质量反馈

当前很多效果判断依赖人工看输出，这对视频项目是必要的，但如果完全依赖肉眼，会带来：

- 调参效率低
- 不同 preset 难量化比较
- 回归优化不容易判断是否真的变好

## 立即推荐的改造项

## 1. Manifest 增量恢复

### 为什么最值得做

这是当前最直接提升稳定性和运行效率的改造。相比继续依赖 `_scan_existing_outputs()` 猜文件，manifest 可以明确记录：

- 本次运行的输入是谁
- 本次运行使用了什么配置
- 哪些 stage 成功产出了哪些文件

恢复时不再“猜”，而是“按记录恢复”。

### 建议做法

为每个输入视频在输出目录写一个 manifest，例如：

```json
{
  "input": {
    "path": "...",
    "mtime": 0,
    "size": 0
  },
  "config_hash": "...",
  "cache_version": 1,
  "stages": {
    "pose_detect": {
      "status": "done",
      "outputs": {
        "keypoints_path": "...",
        "video_info": {}
      }
    },
    "h2v_convert": {
      "status": "done",
      "outputs": {
        "h2v_path": "...",
        "cropped_keypoints_path": "...",
        "h2v_size": [1080, 1920]
      }
    }
  }
}
```

恢复时至少校验：

- input path
- input mtime
- input size
- config hash
- cache version

### 推荐优先接入的 stage

建议首批接入：

1. `pose_detect`
2. `h2v_convert`
3. `body_warp`
4. `color_grade`
5. `ken_burns`

其中 `ken_burns` 应优先纳入，因为它本身耗时较高，缓存收益非常明显。

`beat_flash` 也值得较早接入 manifest，但可按复杂度决定：

- 如果改动很小，可与 `ken_burns` 同批接入
- 如果会牵连更多后处理 stage，可作为第二批优先项

### 预期收益

- 第二次跑同一视频时更稳定地跳过
- 改参数后不容易误用旧缓存
- 对高耗时 stage 的重复运行节省更明显
- 中间文件管理更清晰

## 2. H2V 策略拆分

### 为什么最值得做

`03_h2v_convert.py` 是当前最值得单独治理的重文件之一。它之所以难维护，不是因为逻辑一定错，而是因为“策略”和“执行”混在了一起。

建议保留在 stage 中：

- 输入读取
- FFmpeg 裁切与 concat
- 输出写回

建议抽到 `lib/crop_strategy.py`：

- track 构建
- lead 评分
- frame classify
- segment merge
- 裁切计划

### 建议先抽出的函数

- `build_tracks(...)`
- `score_track(...)`
- `select_lead_track(...)`
- `classify_frame(...)`
- `classify_frames(...)`
- `merge_segments(...)`

### 建议优先配置化的常量

- `top_region_ratio`
- `pano_person_threshold`
- `min_segment_seconds`
- `lead_crop_padding`
- `multi_crop_width_ratio`
- `track_match_threshold`

### 原则

第一版重点是“拆结构”，不是“升算法”。

也就是说：

- 尽量保持当前行为接近原版
- 先把策略逻辑移出去
- 后面再慢慢优化判定质量

### 预期收益

- `03_h2v_convert.py` 会明显变轻
- 后续单独调 H2V 策略更容易
- 更方便做针对性测试
- 不需要为了改策略反复碰 FFmpeg 执行部分

---

## 3. 轻量 Metrics

### 为什么值得顺手做

不需要一开始就做完整 QA 系统，只要先补一份轻量 `run_metrics.json`，就已经能明显提升调参效率。

### 第一版建议记录

- `pose_detect_rate`
- `avg_person_count`
- `lead_center_jitter`
- `output_frame_delta`
- `audio_video_duration_delta`
- `video_duration_sec`

后续可补：

- `ankle_out_of_frame_rate`
- `crop_center_drift`

### 预期收益

- 调参时不再完全依赖主观感觉
- 不同 preset 可结构化比较
- 后续优化 H2V 或缓存策略时有更清晰反馈

## 暂缓的改造项

以下改造不是没有价值，而是当前阶段投入产出比不高，建议暂缓。

### 1. 配置大一统

例如把所有配置全部收口成：

```python
config["stages"]["pose_detect"]["enabled"]
config["stages"]["pose_detect"]["gpu"]
config["stages"]["body_warp"]["leg_lengthen"]
```

这个方向在更大规模的平台型项目里有意义，但当前问题是：

- 迁移工作量大
- preset 文件需要较大改动
- 现有代码已经能跑
- 对当前最痛点的问题帮助不直接

### 2. Stage requires/provides 契约体系

给每个 stage 增加：

- `requires`
- `provides`

这在大型 pipeline 框架里常见，但当前阶段会带来较多 boilerplate，且不是主瓶颈。

### 3. Context 分层

把上下文严格拆成 `runtime / assets / metrics / cache` 理论上更规整，但当前阶段：

- 修改面较大
- 分类边界未必天然清晰
- 收益不如 manifest 和 H2V 拆分直接

## 推荐推进顺序

最推荐的顺序是：

1. 先做 Manifest
2. 尽早把 `ken_burns` 接入 Manifest
3. `beat_flash` 作为同批可选或第二批优先
4. 再拆 H2V 策略
5. 顺手补轻量 Metrics
6. 最后更新文档和整理目录

原因：

- Manifest 先做，能稳住缓存和恢复
- 对耗时大的 `ken_burns` 越早接入缓存，收益越大
- H2V 拆分后，后续调参效率最高
- Metrics 适合用来比较拆分前后的效果

如果只做一轮小规模整理，建议就做下面这五件事：

1. 新建 `pipeline/manifest.py`
2. 首批接入 `pose_detect / h2v_convert / body_warp / color_grade / ken_burns`
3. 新建 `lib/crop_strategy.py`
4. 输出 `run_metrics.json`
5. 更新 `视频生成流程.md` 并整理根目录

## 最终结论

当前项目不适合做一轮“全面框架化重构”，更适合做“高收益、低干扰”的局部重构。

最值得立即做的是：

1. 用 Manifest 稳住增量恢复
2. 把 H2V 策略从重 stage 中拆出来
3. 补一份轻量质量指标输出
4. 更新过时文档
5. 顺手整理根目录与 `.gitignore`

这条路线更务实，也更符合当前项目节奏。
