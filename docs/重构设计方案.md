# 健身视频优化流水线重构设计方案

## 文档目的

这份文档用于沉淀当前项目的重构思路，来源于一次针对代码结构、缓存机制、配置体系、横转竖策略与工程可维护性的专项分析。目标不是重写项目，而是在保留现有 stage 能力与处理效果的前提下，逐步把隐式约定显式化，让系统更稳定、更易扩展、更适合长期维护。

本文重点覆盖：

- 当前项目的主要工程问题
- 重构目标与原则
- 分阶段实施路线
- 各 PR 的具体设计方向
- 推荐的文件拆分与职责调整

## 当前项目判断

项目当前已经具备明确的实用价值，尤其是主流程已经形成：

1. 姿态检测
2. 稳定化
3. 横转竖
4. 运镜
5. 体型调整
6. 调色
7. 导出

这说明项目已从“原型验证”阶段进入“功能堆叠后需要治理”的阶段。现阶段最需要补强的不是再增加一个新特效 stage，而是以下四类基础能力：

1. 配置一致性
2. 增量恢复可信度
3. 阶段输入输出边界
4. 质量评估和自动护栏

## 当前主要问题

### 1. 配置结构不统一

当前代码里同时存在多种配置路径：

- `config["stages"]["xxx"]`
- `config["body_warp"]`
- `config["color_grade"]`
- `config["ken_burns"]`
- `config["pose_gpu"]`
- `config["preview"]`

这会导致以下问题：

- preset 与 CLI override 的落点不一致
- 不同 stage 读取配置方式各异
- 后续新增配置时容易继续扩散
- 文档和代码容易偏离

### 2. stage 之间依赖关系不显式

现在 `PipelineContext` 基本是一个平铺字典容器，stage 之间通过约定俗成的 key 传递数据，例如：

- `keypoints`
- `video_info`
- `h2v_path`
- `cropped_keypoints`
- `warped_path`
- `color_path`

这能跑，但会带来：

- 缺依赖时错误位置靠后
- 难以做 resume-from / only-stage
- 后续维护者不容易快速判断某阶段依赖什么、产出什么

### 3. 增量恢复依赖文件名猜测

当前 engine 会扫描输出目录，依靠已知文件命名来恢复中间产物。这种方式短期有效，但有明显问题：

- 产物命名一旦变化，恢复就不可靠
- `*_new.mp4` 这类 fallback 文件不好纳入推断
- 配置变化后，旧缓存可能被误复用
- 恢复结果与实际本轮参数不一定一致

### 4. H2V 决策逻辑耦合过重

`03_h2v_convert.py` 既负责：

- 输入加载
- 人物追踪
- 领操人识别
- 场景判定
- 分段合并
- FFmpeg 裁切
- cropped keypoints 回写

这会使该文件成为后续演化的阻塞点。只要想改“领操人识别”或“全景判定”，就必须在复杂的视频裁切执行逻辑中穿插修改，风险和维护成本都偏高。

### 5. 质量缺少自动化指标

目前项目主要依赖人工看输出效果来判断是否改好，但对于健身视频这种对人体完整性、主体稳定性、节奏连续性要求较高的场景，更适合引入自动质量指标，例如：

- 关键点检出率
- 人数稳定性
- 领操人中心抖动
- 脚踝/膝盖出框率
- 输出帧数偏差
- 音视频时长偏差

## 重构目标

本次重构建议遵循以下目标：

1. 不重写已有算法能力
2. 不打乱现有 stage 主顺序
3. 优先收口配置、上下文、恢复机制
4. 将复杂决策从 stage 本体中拆出
5. 为后续自动化质量评估留接口

换句话说，这不是一次“推翻重来”的重构，而是一次“骨架矫正 + 关键模块解耦”。

## 重构原则

### 1. 保持行为尽量稳定

在前几个 PR 里，应优先保持输出行为接近当前版本，避免结构调整与算法变动同时发生。

### 2. 隐式约定显式化

例如：

- stage 依赖什么
- stage 产出什么
- 当前配置最终长什么样
- 哪些缓存可复用

都应从“默认靠人记住”变成“代码中有结构化表达”。

### 3. 分阶段提交

建议按多个中小 PR 推进，而不是一次性大改。这样更容易验证，也更方便随时回退。

## 推荐实施路线

建议按 5 个阶段推进：

1. PR-1 配置与装配层收口
2. PR-2 上下文与 stage 契约
3. PR-3 manifest 增量恢复
4. PR-4 H2V 策略拆分
5. PR-5 质量指标与测试样本

## PR-2 上下文与 stage 契约

### 目标

引入更清晰的 `PipelineContext`，并让核心 stage 开始声明依赖与产出。

### 本阶段新增文件

- `pipeline/context.py`
- `stages/base.py`

### 本阶段重点改造文件

- `pipeline/engine.py`
- `stages/01_pose_detect.py`
- `stages/03_h2v_convert.py`
- `stages/04_ken_burns.py`
- `stages/05_body_warp.py`
- `stages/06_color_grade.py`
- `stages/07_export.py`

### `PipelineContext` 建议结构

```python
class PipelineContext:
    input_path: Path
    output_dir: Path
    config: dict

    runtime: dict
    assets: dict
    metrics: dict
    cache: dict
```

### 建议的方法

- `set_asset() / get_asset() / has_asset()`
- `set_runtime() / get_runtime()`
- `set_metric() / get_metric()`

兼容期可保留：

- `set()`
- `get()`
- `has()`

但内部逐步转发到新分层结构。

### 运行时数据归属建议

放到 `runtime`：

- `video_info`
- `h2v_size`
- `ken_burns_ratio`
- `config_hash`
- `run_started_at`

放到 `assets`：

- `keypoints`
- `cropped_keypoints`
- `stabilized_path`
- `h2v_path`
- `warped_path`
- `face_warp_path`
- `color_path`
- `audio_path`
- `final_path`

### `stages/base.py` 建议结构

```python
class BaseStage:
    name = ""
    requires = []
    provides = []

    def describe(self):
        ...

    def run(self, ctx):
        raise NotImplementedError
```

### engine 建议新增能力

运行 stage 前检查：

- `stage.requires`

运行后收集：

- `stage.provides`

典型逻辑：

```python
missing = self._check_stage_requirements(stage, ctx)
if missing:
    raise RuntimeError(...)
```

### 核心 stage 首批契约建议

`PoseDetectStage`

- `name = "pose_detect"`
- `requires = []`
- `provides = ["video_info", "keypoints"]`

`H2VConvertStage`

- `name = "h2v_convert"`
- `requires = ["video_info", "keypoints"]`
- `provides = ["h2v_path", "cropped_keypoints", "h2v_size"]`

`KenBurnsStage`

- `name = "ken_burns"`
- `requires = ["video_info"]`
- `provides = ["ken_burns_path", "ken_burns_ratio"]`

`BodyWarpStage`

- `name = "body_warp"`
- `requires = ["video_info"]`
- `provides = ["warped_path"]`

`ColorGradeStage`

- `name = "color_grade"`
- `requires = ["video_info"]`
- `provides = ["color_path"]`

`ExportStage`

- `name = "export"`
- `requires = ["video_info"]`
- `provides = ["final_path"]`

### PR-2 验收标准

1. 主链仍可跑通
2. 核心 stage 已声明 `name / requires / provides`
3. `video_info` 和中间产物开始按 `runtime / assets` 分层

## PR-3 manifest 增量恢复

### 目标

用 manifest 替代基于文件名猜测的恢复逻辑。

### 本阶段重点改造文件

- `pipeline/manifest.py`
- `pipeline/engine.py`
- `stages/01_pose_detect.py`
- `stages/03_h2v_convert.py`
- `stages/05_body_warp.py`
- `stages/06_color_grade.py`

### manifest 文件建议

建议放在输出目录中，命名为：

- `<video_stem>_manifest.json`

建议结构：

```json
{
  "input": {
    "path": "...",
    "mtime": 0,
    "size": 0
  },
  "config_hash": "...",
  "cache_version": 1,
  "stages": {
    "pose_detect": {
      "status": "done",
      "outputs": {
        "keypoints_path": "...",
        "video_info": {}
      }
    }
  }
}
```

### `pipeline/manifest.py` 建议职责

- `get_manifest_path(ctx)`
- `load_manifest(ctx)`
- `save_manifest(ctx, manifest)`
- `build_input_fingerprint(path)`
- `is_manifest_compatible(ctx, manifest)`
- `restore_context_from_manifest(ctx, manifest)`
- `record_stage_result(ctx, stage_name, outputs)`

### engine 建议改造

运行开始时：

1. 计算 `config_hash`
2. 写入 `ctx.runtime`
3. 尝试读取 manifest
4. 校验 input 指纹和 config hash
5. 兼容时恢复上下文

每个 stage 成功后：

1. 收集 `stage.provides`
2. 记录到 manifest

### 首批接入 manifest 的 stage

建议先接这 4 个最值钱的 stage：

1. `pose_detect`
2. `h2v_convert`
3. `body_warp`
4. `color_grade`

### `pose_detect` 缓存建议补的元数据

- `cache_version`
- `config_hash`
- `input_mtime`
- `model_name`

### manifest 兼容条件建议

至少要求：

- `input.path` 相同
- `input.mtime` 相同
- `input.size` 相同
- `config_hash` 相同
- `cache_version` 相同

任一不一致，则整份 manifest 不恢复。

### PR-3 验收标准

1. 同一输入第二次运行时，关键 stage 能稳定跳过
2. 改动关键配置后，不会误复用旧中间产物
3. 删除单个中间文件后，恢复逻辑不会误判可用

## PR-4 H2V 策略拆分

### 目标

把 `03_h2v_convert.py` 的“策略逻辑”和“执行逻辑”拆开。

### 本阶段重点改造文件

- `lib/crop_strategy.py`
- `stages/03_h2v_convert.py`
- 可选少量改 `pipeline/schema.py`

### 现状中的多重职责

`03_h2v_convert.py` 当前同时负责：

1. 输入读取
2. 人物追踪
3. 领操人识别
4. 场景判定
5. 分段合并
6. FFmpeg 裁切
7. cropped keypoints 写回

### 拆分原则

保留在 stage 中：

- 输入读取
- 输出写回
- FFmpeg 执行

迁移到 `lib/crop_strategy.py`：

- track 构建
- lead score
- frame classify
- segment merge
- 裁切决策计划

### `crop_strategy.py` 第一版建议函数

- `build_tracks(keypoints, total_frames, cfg)`
- `score_track(track, cfg)`
- `select_lead_track(tracks, cfg)`
- `classify_frame(pose_data, lead_track, cfg)`
- `classify_frames(keypoints, tracks, lead_track, total_frames, cfg)`
- `merge_segments(frame_decisions, fps, cfg)`
- `collect_segment_crop_plan(...)`

### `h2v_convert` 推荐新增配置

建议在 schema 中扩展：

```python
"h2v_convert": {
    "enabled": True,
    "target_ratio": 9 / 16,
    "output_width": 1080,
    "output_height": 1920,
    "top_region_ratio": 0.70,
    "lead_crop_padding": 0.15,
    "multi_crop_width_ratio": 0.95,
    "pano_person_threshold": 3,
    "min_segment_seconds": 0.8,
    "track_match_threshold": 0.2,
}
```

### 推荐迁出的硬编码

建议优先把这些常量转为配置：

- `orig_h * 0.70`
- `PANO_PERSON_THRESHOLD = 3`
- `MIN_SEG = max(int(fps * 0.8), 15)`
- `orig_w * 0.95`
- track 匹配阈值 `0.2`
- lead 匹配窗口 `0.15`

### PR-4 的重点

第一版重点是“拆”，不一定立刻“升算法”。建议先保持当前行为尽量接近原版，只把策略搬到独立模块中。

### PR-4 验收标准

1. `03_h2v_convert.py` 明显变短
2. track / lead / classify / merge 已迁到 `crop_strategy.py`
3. 核心常量可通过配置调节
4. 重构前后输出行为大体一致
